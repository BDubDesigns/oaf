# Decision 0008: Build Receipt Format

- **Status:** Accepted
- **Date:** 2026-07-08
- **Decision type:** Receipt format / audit-trail design (Alpha 0)
- **Supersedes:** none
- **Superseded by:** none
- **Related:** `docs/receipts.md`, `docs/doctrine.md`, `docs/sandbox.md`,
  `docs/package-policy.md`, `docs/app-structure.md`, issues #4, #5, #6,
  #7, #8, #10, #21

## Context

OAF's doctrine §8 requires receipts for meaningful work, and decisions
0001, 0003, 0005, and 0006 all say dependency, schema, and command work
must be receipted. But until now there was no defined receipt format. Issue
#10 defines it. A future pretty viewer is scoped to #21. This decision
locks the *shape and philosophy* of receipts before any emitter exists.

## Decision

1. **OAF receipts are structured JSON first.**
2. **Raw receipt JSON is the source of truth.**
3. **Markdown or pretty-UI views are renderers, not canonical storage.**
4. **Receipts are audit trails, not marketing summaries.**
5. **Failures, skipped checks, warnings, and human approvals must remain
   visible** in any rendering.

### Format principles

- Receipts are versioned via `schemaVersion` so the format can evolve
  without breaking old receipts.
- Receipts are stored per-app under `oaf/receipts/` (established by
  decision 0004 and generated by `oaf init`, #8).
- Command-log entries (from the sandbox, decision 0005) are referenced
  inside receipts; receipts contextualize *why* commands ran.
- Package/install changes (decision 0006) are recorded in receipts as
  security-relevant, approval-gated events.
- Secrets are never stored; only names/paths with a `redacted` marker.
- Screenshots are stored as references (path + hash), never raw blobs in
  JSON.
- The schema is useful, versioned, and extensible — deliberately not
  perfect.

### Lifecycle

Status enum: `pending`, `in_progress`, `success`, `partial`, `failed`,
`blocked`, `needs_review`, `cancelled`.

### Relationship to #21

Receipts are JSON-first precisely so the future pretty viewer (#21) can
render them without owning the data. This issue defines the format; #21
defines presentation. The schema must not be coupled to any one renderer.

## Consequences

- Agent work becomes reviewable after the fact without re-execution.
- Dependency/schema/command work is now tied to a concrete, auditable
  artifact.
- Renderers can be built later (#21) against a stable JSON contract.

## Confirmed deferred to later issues

- **Pretty receipt viewer / renderer** → #21.
- **Receipt storage backend / database** → future (non-goal for now).
- **Receipt dashboard, sharing, auth, analytics, search** → non-goals for
  this issue.
- **Exact emitter implementation** (writing receipts from `oaf` commands)
  → future implementation work.
- **Perfect schema migrations** → avoided on purpose; `schemaVersion`
  handles evolution.
